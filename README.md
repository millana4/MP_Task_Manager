# Mercator Task Manager

Сервис управления сбором данных о товарах на маркетплейсах для эконометрического
исследования рынка одежды. Оркеструет парсер Ozon: принимает в csv наборы страт (сеты),
раздаёт задачи подбора и обхода карточек, складывает результаты в две базы —
Postgres (структура, статусы) и ClickHouse (временные ряды замеров и снимки).

## Назначение

Система решает задачу репрезентативного сбора данных о ценах, наличии и
характеристиках одежды. Администратор загружает таблицу страт
(гендер, слой, предмет, поисковый запрос, нужное количество), а сервис наполняет
её реальными карточками через парсер и затем периодически обходит их, накапливая
временной ряд для эконометрического анализа.

Ключевые понятия:

- **Сет** — набор страт для одного сочетания маркетплейс + категория + гео
  (например, одежда на Ozon в Санкт-Петербурге). Единица загрузки.
- **Страта** — строка сета: что искать (`query`), сколько карточек нужно
  (`count`), сезонность и доля базовой коллекции. Единица подбора.
- **Карточка** — конкретный товар (SKU), привязанный к страте. Единица обхода.
- **Замер** — снимок цены/наличия/рейтинга карточки в момент времени. Строка
  временного ряда.

## Архитектура

Сервис оркестрирует задачи по сбору данных между парсерами. Взаимодействие с парсером идёт
асинхронно через Kafka: задачи и результаты — это JSON-сообщения в топиках,
разделённых по гео.

### Потоки данных

В сервисе два независимых потока, каждый со своей парой топиков:

- **Подбор** (`select.tasks.<гео>` / `select.results.<гео>`): по стратам сета
  формируются задачи «подбери N карточек под запрос», парсер возвращает списки
  карточек, сервис пишет их в базы.
- **Обход** (`parse.tasks.<гео>` / `parse.results.<гео>`): по активным карточкам
  формируются задачи «распарсить карточку по SKU», парсер возвращает свежие данные,
  сервис дописывает замер.

Разделение топиков по гео позволяет привязывать парсеры к регионам (парсер в
Санкт-Петербурге читает только `*.spb`).

### Хранилища

- **Postgres** — структура и жизненный цикл: сеты, страты, карточки (SKU, статус,
  счётчик неудач). Схема через Flyway (`db/migration`).
- **ClickHouse** — аналитические данные: `measurement` (частые замеры цен и
  наличия, MergeTree) и `card_snapshot` (медленно меняющиеся данные - название,
  характеристики, варианты тавара, вектор эмбеддинга).

Связь между базами — по `card_id` (UUID, генерируется при записи карточки в
Postgres) и `sku`.

## Запуск

### Инфраструктура

Поднять базы и брокер:

```
docker compose up -d
```

Поднимаются три контейнера: `tm-postgres` (порт 5433), `tm-clickhouse` (8123),
`tm-kafka` (9092).

Схема Postgres применяется автоматически через Flyway при старте приложения.
Схему ClickHouse нужно накатить вручную (один раз):

```
docker exec -i tm-clickhouse clickhouse-client --database taskmanager \
  --password taskmanager --multiquery < src/main/resources/clickhouse/schema.sql
```

Kafka-топики создаются автоматически при старте приложения.

### Приложение

```
./gradlew bootRun
```

Приложение стартует на порту 8080. Требует запущенного парсера Ozon на
`localhost:8010` (настраивается через `PARSER_OZON_URL`).

## API

### Загрузка сета

```
POST /api/v1/sets/import
```

Multipart-форма: `file` (CSV со стратами), `marketplace`, `category`, `geo`.
Создаёт новый сет и его страты. Возвращает `setId` и число страт.

```
curl -X POST http://localhost:8080/api/v1/sets/import \
  -F "file=@set.csv" \
  -F "marketplace=ozon" \
  -F "category=clothing" \
  -F "geo=Санкт-Петербург"
```

### Наполнение сета

```
POST /api/v1/sets/{id}/fill          # синхронно (прямой HTTP к парсеру, для отладки)
POST /api/v1/sets/{id}/fill-async    # асинхронно через Kafka (202 Accepted)
```

Асинхронный вариант отправляет задачи в очередь и сразу возвращается; карточки
приходят и записываются по мере ответов парсера.

### Выдача данных для аналитики

```
GET /api/v1/analytics/cards/{cardId}/measurements   # временной ряд замеров карточки
GET /api/v1/analytics/strata/{stratumId}/cards      # карточки страты
```

## Формат CSV

Полная рабочая таблица со стратами (разделитель — запятая, кодировка UTF-8, с
заголовком). Сервис берёт по именам колонок только нужные семь колонок: 
- гендер (Ж/М), 
- слой («0 нательный» → 0), 
- предмет одежды,
- кол-во карточек для страты, 
- сезонный сплит (TRUE/FALSE), 
- доля базы (только у сезонных), 
- поисковый запрос. 


## Контракт с парсером

Сообщения — JSON в snake_case. Парсер должен:

- **читать** задачи из `select.tasks.<гео>` (поля: `task_id`, `set_id`,
  `stratum_id`, `geo`, `query`, `count`, `is_seasonal`, `base_share`, `exclude`);
- **возвращать** результаты в `select.results.<гео>`, возвращая эхом
  `task_id`, `set_id`, `stratum_id`, `geo` и добавляя `ok`, `requested_count`,
  `found_count`, `cards`, `error`;
- аналогично для обхода: `parse.tasks.<гео>` → `parse.results.<гео>`.

Поле `set_id` — новое по сравнению с HTTP-контрактом парсера: его нужно вернуть в
результате для учёта партии наполнения.

## Учёт партий и надёжность

Наполнение сета отслеживается как **партия**: сервис знает, сколько задач
отправлено и сколько результатов вернулось. Партия закрывается, когда пришли все
результаты либо истёк таймаут (по умолчанию 4 часа, настраивается через
`taskmanager.fill.batch-timeout`). Состояние партий хранится в памяти
(изолировано в `FillBatchRegistry` — при необходимости заменяется на БД).

Неполный сбор встречается штатно (парсер может не найти нужное число уникальных
карточек) и не блокирует систему. Механизм добора использует `exclude` — список
уже собранных карточек, которые парсер не должен повторять.

Необрабатываемые сообщения (битый JSON) уходят в dead-letter топик
`select.results.dlq` для последующего разбора.

## Жизненный цикл карточки

- `active` — карточка отвечает при обходе;
- `stale` — не ответила, счётчик неудач растёт (порог настраивается через
  `taskmanager.card.max-failed-attempts`, по умолчанию 5);
- `dropped` — списана после превышения порога неудач.

Успешный обход сбрасывает счётчик и возвращает карточку в `active`. Выбывшие
карточки восполняются добором.

## Периодический обход

Планировщик раз в час обходит активные карточки, отправляя задачи парсинга и
дописывая свежие замеры. Защита от наложения циклов: новый обход не стартует, пока
не завершён предыдущий.

## Структура проекта

```
src/main/java/com/marketscan/taskmanager/
├── contract/Ozon/     — контракты обмена с парсером (карточка, запрос, ответ)
├── entity/            — JPA-сущности Postgres (сет, страта, карточка)
├── repository/        — репозитории Postgres
├── clickhouse/        — модели и репозитории ClickHouse (замер, снимок)
├── set_csv/           — разбор CSV со стратами
├── client/            — HTTP-клиент парсера (RestClient)
├── kafka/             — продюсеры, консьюмеры, сообщения, маршрутизация по гео
├── service/           — бизнес-логика (импорт, наполнение, учёт партий,
│                         жизненный цикл, планировщики)
├── controller/        — HTTP-эндпоинты
└── config/            — конфигурация (источники данных, RestClient, топики Kafka)
```

## Тесты

Интеграционные тесты используют Testcontainers (Postgres) и живые контейнеры
(ClickHouse, Kafka). Живые тесты, требующие запущенного парсера, помечены
`@Disabled` и запускаются вручную.

```
./gradlew test
```

## Известные ограничения и дальнейшие шаги

- Вектор эмбеддинга снимка пока пустой (место под модель зарезервировано).
- Схема ClickHouse накатывается вручную (не через Flyway).
- Состояние партий — в памяти, не переживает перезапуск приложения.
- Реализован один регион (Санкт-Петербург); добавление региона — новый код в
  `GeoTopics` и пара топиков.
